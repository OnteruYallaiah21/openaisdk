⚡ Trade‑off: higher turns → more reasoning, higher cost/latency. Lower turns → faster, may truncate.

Tools

Trade‑off: parallel → faster but may cause race. Sequential → deterministic, higher latency.
For CPU‑bound or ordered ops, force sequential.

🔬 Inside RunContextWrapper — what it carries

turn_input

The exact input for the current turn (user message or previous LLM/tool result). Used for context.

usage (token breakdown)

  • input_tokens · prompt
  • output_tokens · completion
  • reasoning_tokens · (o1/o3 internal)
  • cached_tokens · prompt caching hit

Trade‑off: enable prompt caching reduces cost/latency, but may affect determinism if cache varies.

tool_input

Stores parameters passed to tool, can be inspected for debugging.

_approvals

Tracks human approval state for tools with approval required. If deactivated, tools run automatically (less safety).

Trade‑off: approvals add latency but prevent risky actions.

Runner turns · if max_turns = 2

Agent will execute up to 2 LLM calls (or tool loops). After 2 turns, it stops and returns final output.
Trade‑off: Turn 2 may include a tool call that won't be processed if limit reached. Use higher turns for multi‑step workflows.

turn_input is updated each turn

CPU‑bound sequential

By default tools run in parallel. For CPU‑intensive or ordered ops, set parallel_tool_calls=False or use run_llm_again to enforce sequential.

Without explicit sequential, LLM may call tools simultaneously.

🔑 Core takeaways
  • LLM decides — Runner executes.
  • RunContextWrapper holds turn_input, usage tokens, approvals.
  • Tools default parallel — force sequential for order/CPU.
  • Guardrails are safety — input/output filters.
  • Handoffs enable multi‑agent.
  • Trade‑offs: max_turns, parallel tools, approvals — balance cost/safety/speed.
