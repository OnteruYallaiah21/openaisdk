Branch Tracking — How Local Branches Map to Remotes

🔗 The Mapping: local branch ↔ remote tracking ref

After setup, you have these mappings. Every local branch can track a remote ref — this is what tells git push / git pull where to go without specifying the remote explicitly.

local: main ══════════▶ origin/main ── Your fork's main (push/pull default)
local: main ── (for sync only) ──▶ upstream/main ── Used with git rebase upstream/main — not auto-tracked
local: feature/llm-router ══════════▶ origin/feature/llm-router ── Your fork's branch (your work pushes here)
local: feat/payment-fix ══════════▶ origin/feat/payment-fix ── Created by: git push -u origin feat/payment-fix
# ─── CHECK: see all tracking relationships ────────────────── git branch -vv * main a3f82bc [origin/main] Add LLM router feature/llm-router c7d9e11 [origin/feature/llm-router] Refactor router feat/payment-fix e4f2b3a [origin/feat/payment-fix: ahead 3] Add validation # "ahead 3" = you have 3 commits not yet pushed to origin # "behind 2" would mean origin has 2 commits you haven't pulled yet # ─── CHECK: see upstream/main commit vs local main ────────── git log --oneline --decorate -8 e4f2b3a (HEAD -> feat/payment-fix) Add validation logic d9c1a2f Fix edge case in payment flow a3f82bc (origin/main, upstream/main, main) Add LLM router b912def Fix agent timeout bug c7180aa Initial agent scaffold # Reading this: # HEAD is on feat/payment-fix (3 commits ahead of main) # main, origin/main, upstream/main all point to a3f82bc → all in sync
⚠️ upstream/main ≠ local main (they are DIFFERENT things)
upstream/main = a read-only pointer stored in .git/refs/remotes/upstream/main. It shows where the company's repo is.
main = your local branch. It won't move just because you fetched.
You must explicitly git rebase upstream/main or git merge upstream/main to move your local main forward.

git fetch — The Big Question: Can I run it from any branch?

✅ YES — you can run git fetch upstream from ANY branch you're on.
It is purely a download operation. It NEVER changes your working directory, NEVER changes your current branch, NEVER merges anything.

📋 Fetch vs Pull vs Pull --rebase — Exact Comparison
Command Working Dir changed? Local branch moved? Merge commit created? History shape When to use
git fetch upstream ❌ No ❌ No ❌ No No change Safely download — peek at what changed. Run anytime.
git merge upstream/main ✅ Yes ✅ Yes ✅ Yes (merge commit M) Non-linear (bubble) Merging long-lived/shared branches. Safer for teams.
git rebase upstream/main ✅ Yes ✅ Yes (replayed) ❌ No Linear (clean) Feature branches before PR. Preferred in most companies.
git pull upstream main ✅ Yes ✅ Yes ✅ Yes Non-linear Quick update on private branch. Creates merge noise.
git pull --rebase upstream main ✅ Yes ✅ Yes ❌ No Linear Best of both: download + rebase in one. Use this daily.
💡 Pro tip: Set this globally once: git config --global pull.rebase true
Now every git pull automatically behaves like git pull --rebase.
04

Merge vs Rebase — Visual Commit History Difference

🔀 git merge upstream/main — creates a merge bubble

BEFORE: upstream/main: A ─── B ─── C ─── D \ your feature: X ─── Y (your commits) AFTER git merge upstream/main: upstream/main: A ─── B ─── C ─── D \ \ your feature: X ─── Y ── M ← merge commit M git log --oneline: M Merge branch 'upstream/main' into feat/payment-fix ← extra noise Y Add validation logic X Fix edge case D Fix token refresh bug (from upstream) C Add LLM router ...
⚠️ Creates a merge commit M that has two parents. History looks like a river delta. Can get messy over many syncs.

✨ git rebase upstream/main — linear, clean

BEFORE: upstream/main: A ─── B ─── C ─── D \ your feature: X ─── Y (your commits) AFTER git rebase upstream/main: upstream/main: A ─── B ─── C ─── D \ your feature: X' ─── Y' ← replayed! git log --oneline: Y' Add validation logic (new hash, same changes) X' Fix edge case (new hash, same changes) D Fix token refresh bug (from upstream) C Add LLM router ...
✅ Your commits X, Y are replayed on top of D (latest upstream). No merge commit. History reads as one clean line. PRs are easier to review.

🛡️ --force-with-lease vs --force:
--force = blindly overwrite remote. Dangerous if a teammate pushed to your branch.
--force-with-lease = only overwrite if remote hasn't moved since your last fetch. Safe. Always use this.

Industry Best Practices — Do vs Don't

✅ Always Do This
✓
Fork only main branch — fetch other branches on demand with git fetch upstream <branch-name>
✓
Set upstream immediately after cloning. Do it before writing any code.
✓
Always work on feature branches, never directly on main.
✓
Rebase before raising PR — clean linear history makes reviews easier.
✓
Use --force-with-lease — never bare --force.
✓
Sync with upstream daily — don't let your fork drift weeks behind.
✓
Clean up branches after merge — delete local + remote feature branches.
❌ Never Do This
✗
Never commit directly to local main — it should always mirror upstream/main.
✗
Never clone the upstream repo directly — clone your fork, add upstream as remote.
✗
Never open PR from your main branch — always from a named feature branch.
✗
Never use bare --force — if remote has new commits, you'll overwrite someone's work.
✗
Never rebase a shared/public branch — rebase rewrites history, breaks others.
✗
Never fork all branches — your fork becomes heavy and unmanageable.
✗
Never raise PR without syncing upstream — stale PRs are painful to review.

git fetch upstream is always safe to run from any branch. Downloads data, never changes your work.

upstream/main ≠ local main. Fetch updates the reference; you still need to rebase to move your branch.
🟡
Rebase > Merge for feature branches. Creates clean, linear history. Required before most PRs.
🔴
--force-with-lease not --force. After rebase, hashes change — you must force-push, but safely.
🍴
Fork only main by default. Fetch extra upstream branches on demand when you need them.
🎯
Never commit to local main. Keep it as a clean mirror of upstream. All work in feature branches.
📋
PR = origin/feat/xyz → upstream/main. Never PR from your main. Never PR without syncing first.
🧹
Clean up after merge. Delete local + remote feature branches. Keeps your repo uncluttered.
